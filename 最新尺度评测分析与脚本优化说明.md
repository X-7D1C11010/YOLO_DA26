## 二、尺度评测结论

### 2.1 `train_26.py` DA 模型 best.pt

`evaluation_scale_grid.jsonl` 对应 `YOLO26s_DA_DANN-4/best.pt`：

| imgsz | mAP50 | mAP75 | mAP50-95 | Precision | Recall |
|---:|---:|---:|---:|---:|---:|
| 512 | 0.4759 | 0.1314 | 0.2012 | 0.5252 | 0.4772 |
| 576 | 0.4670 | 0.1159 | 0.1849 | 0.5064 | 0.5117 |
| 640 | 0.4939 | 0.1412 | 0.2013 | 0.5386 | 0.5325 |
| 704 | 0.4928 | 0.1464 | 0.2041 | 0.5644 | 0.5030 |
| 768 | 0.4637 | 0.1421 | 0.2004 | 0.5908 | 0.4615 |
| 896 | 0.4291 | 0.1660 | 0.1956 | 0.5153 | 0.4715 |

结论：

- DA 模型 mAP50 最佳尺度是 640，mAP50=0.4939。
- 704 与 640 几乎持平，mAP50-95 和 mAP75 略好，Precision 更高，但 Recall 略低。
- 768/896 开始下降，说明继续增加输入尺度不能解决独立集泛化问题。
- 实际部署或后续错误分析建议优先使用 `imgsz=640`；如果更重视框质量和误检抑制，可以同时保留 `704` 作为候选。

F1 曲线观察：

- DA@640：最佳 F1 约 0.53，最佳 confidence 约 0.195。
- DA@704：最佳 F1 约 0.53，最佳 confidence 约 0.232。

这说明低阈值 `conf=0.001` 适合计算 mAP，但不适合部署展示；部署阈值建议从 0.20~0.30 起步。伪标签阈值必须远高于部署阈值，建议从 0.55~0.75 起步，当前不宜使用低阈值伪标签。

## 三、总体技术判断

这次结果进一步确认：

1. 1024/1280 不是当前项目的提分方向。独立集原图虽然是 1024，但高输入尺度会放大 SAR 强散射背景和域差异，导致 FP 增加、Recall/Precision 失衡。
2. 640/704 是 DA 模型当前最有效的推理区间；512/576 是 SAR-only 当前最有效的推理区间。
3. DA 的确有效：在最佳尺度附近，DA 比 SAR-only 高约 8~10 个 mAP50 点。
4. 但 DA 仍未解决背景误检和定位质量问题，mAP50 仍停留在 0.49 左右，距离 0.80 仍需数据与训练策略层面的改动。
5. 现在可以把主要优化重点从“修复数据路径/类别映射”转向“域均衡采样、低尺度稳定训练、hard negative、少量目标域微调”。

## 四、已完成脚本优化

### 4.0 2026-06-26 晚间更新：切换到 24GB/1024 主训练配置

你确认服务器为 24GB 显卡且当前无其他程序占用，因此训练脚本已从 12GB 保守配置切换为 24GB 高显存配置。新的思路是：

- 训练主尺度采用 `imgsz=1024`，用于验证“独立测试集目标框尺度与训练集不一致，训练也应覆盖 1024 尺度”的假设；
- 默认 batch 先使用 16，稳定后再尝试 20；不再默认 24；
- 继续每轮训练/验证结束后主动清理 CUDA 缓存；
- 不恢复旧实验中的高风险组合 `lr0=0.002 + domain_weight=0.02`，因为旧日志从第 25 轮开始出现 EMA NaN/Inf；
- 如果出现 OOM 或 CUDA driver error，优先保持 `imgsz=1024`，降低 batch，并关闭多尺度训练。

#### 4.0.1 关于 `CUDA driver error: invalid argument`

你在 `imgsz=1024,batch=24` 下遇到的报错发生在 `TaskAlignedAssigner.select_candidates_in_gts()`：

```text
bbox_deltas = torch.cat(...).view(bs, n_boxes, n_anchors, -1)
RuntimeError: CUDA driver error: invalid argument
```

这不是普通数据路径错误，而是 1024 大尺度、大 batch、某些多目标图像共同导致 TAL 正样本分配阶段构造了过大的中间张量。1024 输入下 anchor 数约 2 万级，若某个 batch 中最大单图 GT 框数较多，`(bs, n_boxes, n_anchors, 4)` 会非常大，CUDA 可能不报标准 OOM，而直接报 driver invalid argument。

已做两项修复：

1. `yolo_source/ultralytics/utils/tal.py` 中的 `select_candidates_in_gts()` 改为按 anchor 分块计算，降低峰值显存和单个 CUDA kernel 压力；
2. 默认训练配置从 `batch=24,multi_scale=0.10` 改为 `batch=16,multi_scale=0`，先保证 1024 主尺度稳定跑通。

同时 `audit_dataset.py` 已增加 `instances_per_image` 和 `top_instance_examples`，用于排查是否存在单图实例数异常多的样本。

推荐 24GB DA 主实验命令：

```bash
python train_26.py \
  --data dataset_aircraft_da_balanced.yaml \
  --imgsz 1024 \
  --batch 16 \
  --nbs 64 \
  --workers 4 \
  --lr0 0.0007 \
  --domain-weight 0.002 \
  --epochs 120 \
  --patience 30 \
  --multi-scale 0 \
  --name DA_bal_dw002_img1024_b16
```

如果第一轮能稳定跑完，并且显存占用明显低于 20GB，可再尝试：

```bash
python train_26.py \
  --data dataset_aircraft_da_balanced.yaml \
  --imgsz 1024 \
  --batch 20 \
  --nbs 64 \
  --workers 4 \
  --lr0 0.0007 \
  --domain-weight 0.002 \
  --multi-scale 0 \
  --name DA_bal_dw002_img1024_b20
```

训练后必须做全尺度评测，而不是只测 1024：

```bash
python test.py runs/detect/DA_bal_dw002_img1024_b16/weights/best.pt \
  --data test.yaml \
  --imgsz 512,576,640,704,768,896,1024 \
  --project runs/detect/runs/eval_DA_bal_dw002_img1024 \
  --output evaluation_DA_bal_dw002_img1024.jsonl \
  --exist-ok
```

判断标准：

- 如果 1024 训练后在 `640/704/1024` 都提升，说明“训练尺度覆盖 1024”假设成立；
- 如果只在 1024 略有提升，但 640/704 下降，需要考虑多尺度训练或 704 预训练 + 1024 微调；
- 如果整体下降，则说明主要瓶颈不是训练输入尺度，而仍是背景误检、域差异或定位质量。

### 阶段 5：少量目标域样本微调

- 无标注：只做高置信伪标签，阈值建议 `0.55~0.75` 起步，且必须人工抽查；

当前 F1 曲线显示最佳部署阈值约 0.20~0.23，但伪标签阈值不能用这个值。伪标签阈值必须更高，否则会把大量背景强散射误检写入训练集。

## 七、最终判断

最新结果没有推翻之前的主结论，反而更明确了方向：

- 数据类别映射问题已基本解决；
- 当前 mAP50 上限主要来自域差异、误检和定位质量，而不是路径或类别配置错误；
- 大尺度输入不是解法，640/704 才是当前 DA 模型的有效区间；
- 下一轮最值得做的是“域均衡 DA + 较弱对抗损失 + 较低学习率 + 低几何扰动 + 640/704 评测”；
- 若目标仍是 mAP50≥80%，后续必须尽早引入少量目标域 hard case 标注或可靠伪标签，否则纯无监督提升空间有限。
